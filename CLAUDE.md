## Project

## Architecture

Source layout: `src/synapsefs/` with entry point registered as `fsspec.specs` (`syn = synapsefs.synapsefs:SynapseFS`).

**Key modules:**
- `synapsefs.py` ÔÇö `SynapseFS(AbstractFileSystem)`: core filesystem implementation using fsspec. Uses `synapseclient.operations` (store, get, delete) and `synapseclient.models` (File, Folder, Project) rather than the legacy `Entity` API.
- `remote_file.py` ÔÇö `RemoteFile`: proxy around local temp files with `on_close()` callback that triggers Synapse upload
- `utils.py` ÔÇö extracted helpers: mode parsing (`parse_mode`, `normalize_mode`), empty-file padding (`pad_empty_file`), file renaming (`rename_to_target`)

**Path resolution:** Supports Synapse IDs (`syn50545516`), human-readable names, and mixed paths (`syn50545516/SubFolder`). Regex: `r"syn[0-9]+"`. Two modes: rooted (constrained to a project/folder) and rootless (paths must start with a Synapse ID). The `syn://` protocol prefix is stripped by `_strip_protocol()`.

**File I/O flow:** `_open()` Ôćĺ download to temp dir Ôćĺ wrap in `RemoteFile` Ôćĺ on close: add null byte if empty Ôćĺ upload to Synapse Ôćĺ cleanup temp file.

## Conventions

- Commit messages: `[JIRA-KEY] Description` (e.g., `[DPE-1275] Run tests with xdist`)
- `from __future__ import annotations` at top of every source file
- Private methods prefixed with `_`; class constants in `UPPER_CASE`
- Use `synapse_errors(path)` context manager for all Synapse API calls ÔÇö it maps `synapseclient` exceptions (`SynapseFileNotFoundError`, `SynapseHTTPError`) to standard Python exceptions (`FileNotFoundError`, `FileExistsError`)
- Thread-local Synapse clients via `threading.local()` ÔÇö each thread gets its own client with a unique temp cache directory

## Constraints

- Do NOT remove cache purging (`self._local.synapse.cache.purge(after_date=0)`) from the `synapse` property ÔÇö an attempt to replace it with empty cache dirs was reverted (commit `afaf4c1`). The purge prevents stale cached files from being served.
- Do NOT create truly empty files on Synapse ÔÇö Synapse rejects them. The codebase writes a null byte (`\x00`) to empty files before upload. Tests expect "empty" files to have `size=1`.
- Do NOT edit files in `src/fs_synapse.egg-info/` ÔÇö these are generated by setuptools.
- Only three entity types are supported: `File`, `Folder`, `Project`. Tables and other entity types raise `ValueError` ÔÇö because the filesystem abstraction has no way to represent them.

## Testing

- **Markers:** `@pytest.mark.integration` for tests requiring live Synapse access. Non-integration tests run without network.
- **Auth:** Integration tests require `SYNAPSE_AUTH_TOKEN` env var. CI runs integration tests only on ubuntu-latest + Python 3.11. When the token is unset, a `pytest_collection_modifyitems` hook in `tests/conftest.py` auto-skips any test whose `item.keywords` contain `"integration"` ÔÇö i.e. anything carrying the `@pytest.mark.integration` marker (set per-module via `pytestmark` in `tests/integration/test_synapse_fs_integration.py`), plus anything whose class/module path matches that string. Gating happens at collection time rather than via fixture chains, so new integration tests skip correctly without needing to depend on the `auth_token` fixture.
- **Concurrency:** Integration tests use `pytest-xdist -n 4`. Thread-local Synapse clients make this safe.
- **Test isolation:** Each integration test run creates a session-level root folder under `syn50555278`, with per-test subfolders cleaned up via finalizers.
- **Test structure:** `tests/unit/` (no network: `test_remote_file.py`, `test_synapse_fs_unit.py`, `test_utils.py`), `tests/integration/test_synapse_fs_integration.py` (requires auth), `tests/conftest.py` (shared fixtures).
