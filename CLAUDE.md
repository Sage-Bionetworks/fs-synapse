## Project

fsspec adapter for Synapse — exposes Synapse files, folders, and projects through the standard `fsspec` API via `syn://` URLs. Downstream consumers: `orca-recipes`, `py-orca`, `py-dcqc`. Generated with PyScaffold 4.3; version managed by `setuptools_scm`.

## Stack

- Python >=3.11, <3.15 (CI tests 3.11–3.14 on ubuntu + macos)
- `fsspec>=2026.0.0`
- `synapseclient>=4.10.0`
- pytest ~9.0, pytest-xdist, pytest-mock, pytest-cov, hypothesis, nbmake
- tox ~3.0
- pre-commit: black, isort (profile=black), flake8 (max-line-length=88), mypy (strict on `synapsefs`), autoflake, interrogate

## Architecture

Source layout: `src/synapsefs/` with entry point registered as `fsspec.specs` (`syn = synapsefs.synapsefs:SynapseFS`).

**Key modules:**
- `synapsefs.py` — `SynapseFS(AbstractFileSystem)`: core filesystem implementation using fsspec. Uses `synapseclient.operations` (store, get, delete) and `synapseclient.models` (File, Folder, Project) rather than the legacy `Entity` API.
- `remote_file.py` — `RemoteFile`: proxy around local temp files with `on_close()` callback that triggers Synapse upload
- `utils.py` — extracted helpers: mode parsing (`parse_mode`, `normalize_mode`), empty-file padding (`pad_empty_file`), file renaming (`rename_to_target`)

**Path resolution:** Supports Synapse IDs (`syn50545516`), human-readable names, and mixed paths (`syn50545516/SubFolder`). Regex: `r"syn[0-9]+"`. Two modes: rooted (constrained to a project/folder) and rootless (paths must start with a Synapse ID). The `syn://` protocol prefix is stripped by `_strip_protocol()`.

**File I/O flow:** `_open()` → download to temp dir → wrap in `RemoteFile` → on close: add null byte if empty → upload to Synapse → cleanup temp file.

## Conventions

- Commit messages: `[JIRA-KEY] Description` (e.g., `[DPE-1275] Run tests with xdist`)
- `from __future__ import annotations` at top of every source file
- Private methods prefixed with `_`; class constants in `UPPER_CASE`
- Use `synapse_errors(path)` context manager for all Synapse API calls — it maps `synapseclient` exceptions (`SynapseFileNotFoundError`, `SynapseHTTPError`) to standard Python exceptions (`FileNotFoundError`, `FileExistsError`)
- Thread-local Synapse clients via `threading.local()` — each thread gets its own client with a unique temp cache directory

## Constraints

- Do NOT remove cache purging (`self._local.synapse.cache.purge(after_date=0)`) from the `synapse` property — an attempt to replace it with empty cache dirs was reverted (commit `afaf4c1`). The purge prevents stale cached files from being served.
- Do NOT create truly empty files on Synapse — Synapse rejects them. The codebase writes a null byte (`\x00`) to empty files before upload. Tests expect "empty" files to have `size=1`.
- Do NOT edit files in `src/fs_synapse.egg-info/` — these are generated by setuptools.
- Only three entity types are supported: `File`, `Folder`, `Project`. Tables and other entity types raise `ValueError` — because the filesystem abstraction has no way to represent them.

## Testing

- **Markers:** `@pytest.mark.integration` for tests requiring live Synapse access. Non-integration tests run without network.
- **Auth:** Integration tests require `SYNAPSE_AUTH_TOKEN` env var. CI runs integration tests only on ubuntu-latest + Python 3.11.
- **Concurrency:** Integration tests use `pytest-xdist -n 4`. Thread-local Synapse clients make this safe.
- **Test isolation:** Each integration test run creates a session-level root folder under `syn50555278`, with per-test subfolders cleaned up via finalizers.
- **Test structure:** `tests/unit_tests.py` (no network), `tests/integration_tests.py` (requires auth). `demos/` (Jupyter notebook tested via `nbmake`).
